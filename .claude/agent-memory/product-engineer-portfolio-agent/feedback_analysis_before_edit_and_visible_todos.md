두 가지 확인된 작업 방식:

1. **분석 먼저, 편집은 나중** — 큰 변경 전에 audit/plan 문서를 먼저 작성하고, 그 안에서 스스로 사실관계를 점검한 뒤 구현으로 넘어가는 순서를 사용자가 명시적으로 요청함(이번 대화에서 "이 두 문서를 완성할 때까지 전체 코드 리팩토링을 하지 마"). 이전 세션(다른 에이전트, human-resource-consultant-and-web-senior-engineer)에서도 동일한 선호가 확인된 바 있음.
2. **미확인 사실은 눈에 보이는 TODO로** — `[TODO: ...]` 텍스트를 실제 렌더링되는 페이지에 `.todo-flag`(점선 밑줄, 이탤릭, muted 색상) 스타일로 노출시키는 방식을 이번에 채택함. 코드 주석으로 숨기지 않고 시각적으로 표시해, 사용자가 무엇을 확인/입력해야 하는지 페이지만 봐도 알 수 있게 함.

**Why**: 이력서/포트폴리오는 개인 브랜딩이 걸린 문서라 되돌리기 부담이 크고, 사실이 아닌 성과·지표·프로젝트를 만들어내면 신뢰도에 직접 타격. 화면에 보이는 TODO는 "배포 전에 반드시 채워야 할 것"을 놓치지 않게 하는 안전장치.

**How to apply**: 앞으로 이 저장소에서 콘텐츠를 추가/수정할 때, (a) 범위가 크면 먼저 분석 문서를 쓰고 합의 후 진행, (b) 사실 확인이 안 된 내용을 넣어야 할 때는 `.todo-flag` 클래스(이미 CSS 정의됨, `index.html` 참고)를 재사용해 화면에서 보이게 표시할 것. 사용자가 실제 정보를 채운 뒤에는 `.todo-flag` span과 대괄호 TODO 텍스트를 제거하고 실제 문장으로 교체.
