# 검색되지 않는 사내 지식 아카이브, 로컬 RAG로 찾아보기 (PoC 회고)

## 개요

사내 PMS로는 오픈소스 이슈 트래커인 Redmine을 사용하고 있다. 거의 10년 가까운 이력이 쌓여 있어 사실상 팀에 남아있는 유일한 참고 기록 아카이브인데, 정작 필요할 때 원하는 내용을 찾기가 어려웠다.

정형화된 형식으로 기록된 게 아니라, 그때그때 발생한 일을 남기기 위한 용도로 이슈가 발행되다 보니 제목과 내용의 일관성이 낮고, 키워드 검색만으로는 과거 이력을 제대로 찾아내기 어려운 상태였다. 있는데 못 쓰는 아카이브인 셈이다.

이 문제를 개인적으로 검증해보고 싶어서, Redmine 이슈를 벡터 검색 기반으로 찾아볼 수 있는 로컬 RAG(Retrieval-Augmented Generation) PoC를 짧게 구성해봤다.

- 기간: 약 하루
- 형태: 개인 PoC, 프로덕션 미배포
- 코드: [github.com/dev-jhjoo/redmine-local-rag-poc](https://github.com/dev-jhjoo/redmine-local-rag-poc)

---

## 왜 로컬로 구성했나

Redmine에 쌓인 이슈는 비공개 사내 데이터다. 검증해보고 싶다는 이유로 이 데이터를 외부 AI API로 보내는 건 처음부터 선택지에 두지 않았다. 그래서 임베딩과 LLM 추론을 전부 로컬에서 돌리는 방향으로 설계했다.

- 임베딩 모델: `nomic-embed-text` (Ollama)
- LLM: `qwen2.5:3b` (Ollama)
- 벡터 DB: Qdrant
- 전체 스택: Docker Compose로 구성해 로컬에서 바로 실행

외부로 데이터가 한 바이트도 나가지 않는 구조라, 비공개 데이터를 다루는 실험치고는 부담 없이 진행할 수 있었다.

## 파이프라인

```text
Redmine Issues (CSV 수동 Export)
  → JSON 변환
  → Embedding 생성 (nomic-embed-text)
  → Qdrant 저장
  → 로컬 LLM 질의응답 (qwen2.5:3b)
```

Redmine 화면에서 이슈를 CSV로 export한 뒤, 스크립트로 JSON 변환 → 임베딩 생성 → Qdrant 인덱싱까지 거치면, CLI에서 자연어로 질의할 수 있는 구조다.

```bash
python scripts/ask.py "Kafka timeout 관련 이슈 찾아줘"
python scripts/ask.py "특정 배치 컴포넌트 장애 원인 관련 이슈 정리해줘"
```

정확한 키워드를 몰라도, 의미가 비슷한 과거 이슈를 찾아낼 수 있을 거라는 게 이 PoC의 가설이었다.

## 결과와 한계

일단 "키워드를 정확히 몰라도 의미 기반으로 검색이 된다"는 방향성 자체는 확인할 수 있었다. 하지만 실제로 써보니 한계가 뚜렷했다.

**이슈 제목만 임베딩한 게 가장 큰 한계였다.** 본문까지 포함하지 않고 제목만 벡터화하다 보니, 제목만으로는 맥락이 부족한 이슈는 검색 정밀도가 눈에 띄게 떨어졌다. 애초에 제목 자체가 일관되지 않게 작성된 이슈가 많았던 원인 문제가, 검색 단계에서도 그대로 한계로 이어진 셈이다.

**조직적으로 도입하기에는 하드웨어 리소스 여력이 없었다.** 로컬 LLM/임베딩을 상시로 돌리려면 별도의 서버 리소스가 필요한데, 이걸 사내에 정식으로 투입할 여력까지는 없어서 개인 PoC 선에서 마무리했다.

## 배운 점

- 비공개 데이터를 다루는 AI 실험에서는, 처음부터 "외부로 데이터를 보내지 않는 구조"를 전제로 설계하는 게 마음 편하게 검증을 진행하는 데 중요하다는 걸 체감했다. 로컬 임베딩·로컬 LLM 조합만으로도 이 정도 파이프라인은 충분히 구성 가능했다.
- 검색 품질은 결국 "무엇을 임베딩하느냐"에 달려 있다. 제목만으로는 한계가 명확했고, 본문까지 포함한 인덱싱이 있어야 실사용 가능한 수준이 될 것 같다.
- 좋은 검색 시스템을 만드는 것보다, 애초에 기록을 정형화해서 남기는 습관이 더 근본적인 해결책일 수 있다는 것도 다시 느꼈다. 검색은 기록이 잘 쌓여 있어야 의미가 있다.

## 다음에 다시 해본다면

- 이슈 본문까지 포함한 인덱싱
- Redmine API 연동으로 수동 CSV export 없이 자동 동기화
- 사내 정식 도입을 전제로 한다면, 로컬 LLM을 상시 구동할 리소스 확보 방안부터 검토
